# Release NitroTron3

End-to-end checklist for cutting a versioned GitHub release. Tag the repo, build the deliverables, gather the GPL-required attribution, write release notes, and push the assets up. The skill never tags or publishes without explicit user confirmation — releases are visible to others and hard to retract cleanly.

## Inputs the skill expects

- A version string in the form `vX.Y` or `vX.Y.Z` (semver-ish; the script doesn't enforce a particular scheme).
- A clean working tree on `main`. If `git status` is dirty, stop and report — never tag a dirty state.

## Steps

### 1. Sanity-check the working tree

```sh
git status            # must be clean
git log -1 --oneline  # show HEAD that will be tagged
```

If dirty, stop. If on the wrong branch, stop. Confirm with the user before continuing.

### 2. Build the firmware fresh

```sh
make clean
make 2>&1 | tail -10
```

Read off the FLASH usage. If the build fails, stop.

### 3. Build the user manual PDF

The manual embeds the per-mode pedal-layout SVGs, which are **generated** from
`docs/gen_layout_svg.py` (see the `update-controls` skill). **Regenerate them
first and confirm the labels match the current controls** — the Makefile only
rebuilds the SVGs when the *script* is newer, so stale labels baked into the
script are NOT caught by `make` and will silently ship in the PDF.

```sh
python3 docs/gen_layout_svg.py        # regenerate from the source of truth
git status --short docs/pedal-mode-*.svg   # did any labels change? eyeball them
make manual
```

The PDF is at `docs/USER_MANUAL.pdf`. If any SVG label is wrong, fix the `MODES`
dict in `docs/gen_layout_svg.py` (never hand-edit the `.svg`), regenerate, and
rebuild. Cross-check the diagrams against the README controls table before
shipping.

### 4. Stage the release artifacts under `release/<version>/`

```sh
mkdir -p release/<version>
cp build/NitroTron3.bin  release/<version>/NitroTron3-<version>.bin
cp build/NitroTron3.hex  release/<version>/NitroTron3-<version>.hex
cp docs/USER_MANUAL.pdf  release/<version>/USER_MANUAL.pdf
cp INSTALL.md            release/<version>/INSTALL.md
cp LICENSE               release/<version>/LICENSE
```

The `release/` directory is gitignored (artifacts live on the GitHub release, not in the repo).

### 5. Assemble `THIRD_PARTY_LICENSES.md`

GPL v3 requires keeping the attribution of every component statically linked into the firmware. Concatenate:

- `lib/HothouseExamples/LICENSE` — Hothouse hardware proxy + examples
- `lib/HothouseExamples/libDaisy/LICENSE` — libDaisy
- `lib/HothouseExamples/DaisySP/LICENSE` — DaisySP
- `src/clouds/LICENSE` or note inside `src/clouds/` — vendored Mutable Instruments Clouds reverb (MIT)

Header for each section: which component it covers and which part of the firmware it contributes to (e.g. "src/clouds/* — Mode B wet-path reverb"). Write the assembled file to `release/<version>/THIRD_PARTY_LICENSES.md`.

### 6. Compute checksums

```sh
cd release/<version>
shasum -a 256 NitroTron3-<version>.bin > NitroTron3-<version>.bin.sha256
shasum -a 256 NitroTron3-<version>.hex > NitroTron3-<version>.hex.sha256
cd -
```

### 7. Draft the release notes

Write `release/<version>/RELEASE_NOTES.md`. Structure:

```markdown
# NitroTron3 <version>

## Highlights
- Plain-English summary of what's new vs. the previous release.

## How to install
See `INSTALL.md` in the assets. TL;DR: hold both footswitches 2 s on the
pedal to enter DFU, then `dfu-util -a 0 -s 0x08000000:leave -D
NitroTron3-<version>.bin`. (Other Hothouse pedals use FS1 alone — not
this one.)

## What's in this release
- `NitroTron3-<version>.bin` — DFU-flashable firmware image
- `NitroTron3-<version>.hex` — Intel HEX (for ST-Link / Daisy Web Programmer)
- `USER_MANUAL.pdf` — pedal reference
- `INSTALL.md` — flashing instructions
- `LICENSE` — GPL v3
- `THIRD_PARTY_LICENSES.md` — attributions for bundled code (libDaisy,
  DaisySP, Hothouse, MI Clouds)
- `*.sha256` — checksums
- `Source code (zip)` / `(tar.gz)` — auto-generated by GitHub from the
  tag

## GPL notes
- NitroTron3 is licensed under GPL v3. Corresponding source is the
  exact tag this release was built from; clone with `--recurse-submodules`.
- *Installation Information* (GPL v3 § 6): see `INSTALL.md` — flashing
  a modified firmware uses the same DFU path described there. No
  keys, no signing, no Tivoization.
```

Show the draft to the user. Edit until they're satisfied.

### 8. Tag and push (after explicit user confirmation)

```sh
git tag -a <version> -m "NitroTron3 <version>"
git push origin <version>
```

**Do not run these without the user explicitly saying "tag and push" or equivalent.** Tags published to remotes are awkward to remove — `git push --delete` works but only before anyone else pulled.

### 9. Create the GitHub release

If `gh` CLI is available:

```sh
gh release create <version> \
  --title "NitroTron3 <version>" \
  --notes-file release/<version>/RELEASE_NOTES.md \
  release/<version>/NitroTron3-<version>.bin \
  release/<version>/NitroTron3-<version>.hex \
  release/<version>/NitroTron3-<version>.bin.sha256 \
  release/<version>/NitroTron3-<version>.hex.sha256 \
  release/<version>/USER_MANUAL.pdf \
  release/<version>/INSTALL.md \
  release/<version>/LICENSE \
  release/<version>/THIRD_PARTY_LICENSES.md
```

If `gh` isn't installed, tell the user to upload via the web UI: github.com → Releases → Draft a new release → pick tag `<version>` → drag the files from `release/<version>/`. Keep the auto-generated `Source code (zip/tar.gz)` assets — they satisfy the GPL's "corresponding source must accompany the binary" clause.

### 10. Verify and report

```sh
gh release view <version>
```

List the assets, the source-code-zip URL, and the tag. Confirm to the user that the release is live and check that the `.bin` checksum matches what the local file produces.

## Rules

- **Never tag or push without explicit user confirmation.** Same for `gh release create`.
- **Never overwrite an existing tag** unless the user explicitly asks for it (`git tag -f` is destructive in a way that's easy to miss in shared history).
- **Don't include `release/<version>/` artifacts in any git commit.** That directory must stay gitignored — large binaries inside a source-code repo are nobody's friend.
- **Verify the binary you upload matches the source at the tag.** Run `make clean && make` at the tag and check the `.bin` checksum before publishing.
- **Skip nothing in `THIRD_PARTY_LICENSES.md`.** Every statically-linked library needs its license text included, even if the linker discarded most of its code.
