Stutter Implementation Overview

NitroTron3 Mode B — Direct-Texture Mode (K2 fully CCW)

Diagnostic document for click debugging. Created 2026-04-24.

1. Signal Flow

When K2 is fully CCW (k2 < 0.02), the grain engine is bypassed. The signal takes this path:

dry (live input)
→
stutter decision
→
wet output
→
Texture Shaper (SW1=MID: bypass)
→
Wet HPF 150 Hz
→
Mix (K6)
→
output
Key observation: With SW1 in MID, the texture shaper is a no-op. The only processing on wet is the HPF before it reaches the mix. The HPF has persistent state — if it's causing transients, it would affect every stutter event identically regardless of our envelope approach.

2. Two States: Writing vs Stuttering

The stutter alternates between two mutually exclusive states:

StateBufferOutput (wet)Trigger
Not stuttering Writing dry into stutter_buf every sample wet = dry (clean passthrough) Random dice roll per sample: RandFloat() < stutter_prob
Stuttering NOT writing (buffer frozen to protect captured chunk) wet = dry × (1−fade) + looped × fade Countdown: stutter_remaining-- until 0

3. The Event Envelope

Each stutter event has a trapezoidal envelope called event_fade:

event_fade over time (192-sample fade = 4 ms at 48 kHz)

The output formula during a stutter event:

wet = dry × (1 − event_fade) + looped × event_fade

At event_fade = 0: wet = dry (clean)
At event_fade = 1: wet = looped (full stutter playback)
In between: a crossfade between the two signals
THE CLICK PROBLEM: dry is the live bass signal. looped is audio captured 50–200 ms ago. These two signals are at completely different phases and possibly different amplitudes. Crossfading between them creates an audible artifact — even over 4 ms — because the blend produces waveform interference at the transition.

4. What the Crossfade Actually Looks Like

Event START: dry fades out, looped fades in
Event END: looped fades out, dry fades in
Visual evidence: The red (dry) and blue (looped) signals are at different phases. During the crossfade region, the blended output (white) shows a bump/dip where the two signals interfere. This is the click.

5. Loop Wrap (Multi-Rep Events)

When an event has multiple repetitions (reps > 1), the read position wraps back to the start of the captured chunk:

Loop playback with 2 reps — raw discontinuity at wrap point

At the wrap point, the last sample of the chunk jumps directly to the first sample. This is a second source of clicks, independent of the event crossfade.

The baked seam crossfade (lines 478–487) attempts to fix this by fading the first/last 32 samples of the chunk to zero at trigger time. But this modifies the buffer, which corrupts it for future events if they trigger before the write head overwrites the modified region.

6. Cut-Out Events

When stutter_is_cutout = true, the event fades to silence instead of playing the buffer:

Cut-out event (10–50 ms): dry signal fades to silence and back

Cut-outs use wet = dry × (1 − event_fade). This should be click-free because it's fading a single signal to/from zero — no crossfade between mismatched signals.
